## Abstract

This editorial presents a solution to the Autocorrect task from AICC Round 1. The objective is to build a character-level autocorrect model that corrects misspellings in text, operating under strict constraints: CPU-only inference, a 250-second evaluation limit on 2,643 test samples, and no pre-trained models. The winning approach achieves a Character Error Rate (CER) of **0.0362** - a substantial improvement over the baseline of 0.0944 - through three key innovations: (1) a learning curriculum that pre-trains on autoencoding before supervised correction, (2) a chunking strategy that decomposes long sequences into overlapping 256-character segments to maximize data utilization, and (3) a bidirectional GRU architecture that exploits the local nature of character-level errors while respecting computational constraints. We provide a detailed analysis of each design decision, including architectural comparisons, training dynamics, and inference optimizations.

## 1. Introduction

The Autocorrect task addresses a fundamental problem in natural language processing: given a text containing typographical errors, produce the intended correct version. Unlike many NLP tasks that require global semantic understanding (sentiment analysis, summarization, translation), autocorrect is inherently **local**: each character's correction depends primarily on its immediate neighbors rather than the document's overall meaning.

The competition's constraints reflect real-world deployment scenarios:
- **No pre-trained models** prohibits the use of transformer-based language models (BERT, GPT, etc.), forcing participants to train from scratch.
- **CPU-only inference** with a **250-second limit** imposes strict computational budgets, making model efficiency paramount.
- The evaluation metric, **Character Error Rate (CER)**, measures the Levenshtein distance between predictions and ground truth at the character level.

This editorial examines the solution submitted by Walnit, which achieved a CER of 0.0362, a 61.6% relative improvement over the baseline. We analyze the architectural, data, and training choices that enable this performance under the given constraints.

## 2. Problem Definition

### 2.1 Task Formulation

Given a misspelled input sequence $X = (x_1, x_2, \dots, x_T)$ where each $x_i$ is a character token, the model must produce a corrected sequence $\hat{Y} = (\hat{y}_1, \hat{y}_2, \dots, \hat{y}_T)$ of identical length $T$. The training data consists of pairs $(X_k, Y_k)$ where $Y_k$ is the ground-truth corrected text.

### 2.2 Evaluation Metric

Performance is measured by **Character Error Rate (CER)**, defined as:

$$
\text{CER} = \frac{S + D + I}{N}
$$

where $S$ is the number of character substitutions, $D$ is the number of deletions, $I$ is the number of insertions, and $N$ is the number of characters in the reference text. CER is computed using the Levenshtein distance at the character level. A lower CER indicates better performance; the baseline solution achieves 0.0944.

### 2.3 Constraints

The task imposes three critical constraints:

1. **Architecture Restriction:** No pre-trained models, APIs, or fine-tuned open-source models. All parameters must be trained from scratch within the notebook execution.
2. **Data Restriction:** No unsupervised learning on the test set; statistics or representations derived from test samples are prohibited.
3. **Hardware Restriction:** Inference must run **CPU-only** and complete evaluation of all 2,643 test samples within **250 seconds**.

## 3. Data and Tokenization

### 3.1 Dataset Characteristics

The dataset consists of pairs of misspelled and corrected English text. A critical observation - and the first key insight - that **the corrected text always has the same number of characters as the misspelled text**. This invariance implies that the task does not require a sequence-to-sequence (encoder-decoder) architecture; an **encoder-only** model that maps each input character to an output character suffices. This eliminates the need for autoregressive decoding, attention mechanisms, or beam search - all of which would be prohibitively expensive under the CPU time constraint.

A second observation is that over 25% of sequences exceed the default truncation length of 768 characters. This necessitates a strategy for handling long sequences without discarding training data.

### 3.2 Character-Level Tokenization

The solution employs a character-level tokenizer built with the Hugging Face `tokenizers` library:

```text
Tokenizer: WordLevel(unk_token="<UNK>")
Pre-tokenizer: Split("", "isolated")
Vocab size: 1000
Special tokens: <PAD>, <UNK>, <SOS>, <EOS>
```

Splitting on the empty string `""` with the `"isolated"` mode ensures that every individual character becomes a separate token. This is appropriate for autocorrect because errors manifest at the character level (insertions, deletions, substitutions of individual characters), and the model must learn to map each input character position to a corrected output character.

### 3.3 Chunking: The Key Data Strategy

A defining innovation of this solution is the **chunking strategy** for training. Rather than truncating long sequences to a fixed maximum length, the tokenizer is configured with:

```python
max_length = 256
stride = 128
return_overflowing_tokens = True
```

This produces overlapping chunks of 256 characters with a 128-character stride, effectively creating $N$ training examples from a sequence of length $L$ where:

$$
N \approx 1 + \left\lfloor \frac{L - 256}{128} \right\rfloor
$$

A sequence of length 768, for example, produces 5 overlapping chunks (offsets 0, 128, 256, 384, 512). This strategy offers three advantages:

1. **Increased training data:** Each long sequence generates multiple training examples, improving sample efficiency.
2. **Local focus:** By restricting each training example to 256 characters, the model is forced to learn local correction patterns rather than relying on global context, which aligns with the task's inherent locality.
3. **Computational efficiency:** Shorter sequences (256 vs. 768) reduce per-iteration time and memory, enabling larger batch sizes and faster training.

Crucially, chunks are computed independently for both the `misspell` and `text` columns, ensuring that the chunk boundaries align identically for inputs and targets. Both chunks are padded to 256 characters, and a length tensor is maintained for each.

### 3.4 Validation and Test Tokenization

Validation and test sets are tokenized differently. For the validation set, sequences are padded to a maximum of 768 characters without chunking, preserving the original sequence boundaries for accurate evaluation. For the test set, sequences are padded to 4096 characters to accommodate the longest possible inputs.

## 4. Model Architecture

### 4.1 Architectural Rationale

The architectural choice is governed by three competing factors: representational capacity, computational cost, and the local nature of the task. While transformers dominate modern NLP, their self-attention mechanism computes pairwise interactions across the entire sequence - overkill for character-level corrections. Recurrent architectures, on the other hand, process tokens sequentially and naturally capture local dependencies.

The solution employs a **bidirectional GRU** - a deliberate departure from the baseline's unidirectional LSTM. Below we justify each component.

### 4.2 Embedding Layer

The embedding layer maps each of the 1,000 character tokens to a 128-dimensional vector:

```python
self.embedding = nn.Embedding(vocab_size, embed_size)
```

An embedding dimension of 128 may seem modest, but character-level embeddings require far less capacity than word-level embeddings. Characters carry limited semantic information in isolation - their meaning emerges from their position and context within a sequence. A 128-dimensional space provides ample room for the model to learn character-level distinctions (vowels vs. consonants, letters vs. punctuation, upper vs. lower case).

### 4.3 Bidirectional GRU

```python
self.gru = nn.GRU(
    embed_size,        # 128
    hidden_size,       # 256
    num_layers=2,
    batch_first=True,
    dropout=0.2,
    bidirectional=True
)
```

**Why GRU over LSTM?** The Gated Recurrent Unit (GRU) simplifies the LSTM architecture by merging the forget and input gates into a single "update gate" and combining the cell state and hidden state. This reduces the parameter count from $3n^2$ (LSTM, four gates) to approximately $2n^2$ (GRU, three gates) per layer. For this task, the GRU's reduced capacity does not hurt performance because character-level corrections are locally determined: the GRU's reset gate, which controls how much of the past hidden state to forget, is particularly well-suited for this: when encountering a misspelled character, the model should reset its context and focus on the surrounding characters rather than maintaining a long-range linguistic state.

**Why bidirectional?** Character corrections benefit from both left and right context. Consider the misspelling "teh" - the correction to "the" requires knowing that the character following "e" is "h". A unidirectional RNN processes left-to-right and would see "t", then "e", and only then "h"; by the time it learns about "h", it has already committed to an output for "e". A bidirectional GRU processes the sequence in both directions simultaneously, concatenating the forward and backward hidden states. This provides symmetric context: for any character position, the model has access to its left and right neighbors.

**Why 2 layers?** Stacking recurrent layers increases the model's temporal range, since each additional layer can capture dependencies at a higher level of abstraction. However, recurrent layers cannot be parallelized across time steps (each step depends on the previous step's hidden state), so deeper models are proportionally slower. Two layers strike a balance: they provide sufficient depth to model character-level patterns without incurring the computational cost of three or more layers.

### 4.4 Output Projection

```python
self.fc = nn.Linear(hidden_size * 2, vocab_size)
```

The bidirectional GRU produces a 512-dimensional vector at each position (256 from the forward pass, 256 from the backward pass). A linear projection maps this to the 1,000-dimensional vocabulary space. The model's output at each position is a probability distribution over character tokens; the corrected character is selected via argmax.

### 4.5 Conditional Sequence Packing

The forward method includes support for `pack_padded_sequence`:

```python
if lens is None:
    out, hidden = self.gru(x, hidden)
else:
    packed = pack_padded_sequence(x, lens, batch_first=True, enforce_sorted=False)
    packed_outputs, hidden = self.gru(packed, hidden)
    out, _ = pad_packed_sequence(packed_outputs, batch_first=True)
```

Packing sequences is a PyTorch technique that prevents the model from processing padding tokens, which wastes computation and biases the hidden state. During validation (sequences padded to 768), padding constitutes a small fraction of total tokens, so the overhead of packing is not justified. During test inference (padded to 4096 with variable-length sequences), packing provides significant computational savings - a critical advantage given the 250-second CPU limit.

## 5. Training Methodology

### 5.1 Three-Phase Training Strategy

The most distinctive aspect of this solution is its **three-phase training curriculum**, which progressively exposes the model to longer contexts and more complex tasks.

<PhaseTable />

### 5.2 Phase 1: Autoencoding Pretraining

In the first phase, the model is trained on an **autoencoding** objective: the correct text (labels) is fed as input, and the misspelled text is the target. Concretely, the loss is:

```python
outputs, _ = model(labels)  # feed correct text as input
loss = criterion(outputs.view(-1, vocab_size), input_ids.view(-1))  # predict misspelled text
```

This formulation might seem counterintuitive: why train the model to predict errors? The purpose is **initialization**. The autoencoding task forces the model to learn the statistical structure of language: character co-occurrence patterns, common letter sequences, and the distribution of characters in English text. By reconstructing the correct input, the model's hidden states come to encode linguistically meaningful representations. When Phase 2 begins, these representations provide a far better starting point than random initialization.

The autoencoding objective is feasible for three epochs (chosen empirically) on the chunked 256-character sequences with a batch size of 1024.

### 5.3 Phase 2: Chunked Correction Learning

Phase 2 performs the primary supervised task: learning the mapping from misspelled text to corrected text on the chunked 256-character sequences.

```python
outputs, _ = model(input_ids)  # feed misspelled text
loss = criterion(outputs.view(-1, vocab_size), labels.view(-1))  # predict correct text
```

The AdamW optimizer is used with a learning rate of $4 \times 10^{-3}$ and weight decay of $10^{-4}$:

```python
optimizer = optim.AdamW(model.parameters(), lr=learning_rate, weight_decay=1e-4)
```

AdamW decouples weight decay from the adaptive gradient updates, providing more effective regularization than standard Adam. Gradient clipping at a max norm of 1.0 prevents catastrophic gradient events, which are common when training recurrent networks from scratch.

Mixed-precision training is employed via `torch.amp.autocast` and `GradScaler`:

```python
scaler = torch.amp.GradScaler("cuda")
with torch.autocast(device_type="cuda"):
    outputs, _ = model(input_ids)
    loss = criterion(outputs.view(-1, vocab_size), labels.view(-1))
scaler.scale(loss).backward()
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
scaler.step(optimizer)
scaler.update()
```

While mixed precision is most beneficial on GPU (the training hardware), it also reduces memory bandwidth on CPU; the patterns established during GPU training (loss scaling, gradient clipping) carry over to the final model.

### 5.4 Phase 3: Full-Length Fine-Tuning

After 18 total epochs of chunked training, the model has mastered local character corrections. However, it has never seen a sequence longer than 256 characters. To generalize to the full-length sequences encountered during evaluation, Phase 3 retokenizes the entire training set **without chunking**, using the same configuration as the validation set (padding to 768). The learning rate is reduced to $1 \times 10^{-3}$, and the batch size decreases from 1024 to 256 (since sequences are longer). Five epochs of fine-tuning adapt the model to operate on full-length inputs.

This curriculum mirrors principles from **progressive learning** in neural networks: start with a simpler task (autoencoding) on shorter sequences, gradually increase task complexity to supervised correction, and finally increase sequence length. Each phase builds on the representations learned in the previous phase.

## 6. Inference

### 6.1 Test-Time Processing

For test inference, sequences are padded to a maximum of 4096 characters and processed with pack_padded_sequence for efficiency:

```python
def wrap_model_prediction(model, batch, device):
    input_ids = batch['input_ids'].to(device)
    lengths = batch['input_lengths'].to(device)
    output = model(input_ids, lengths)[0]
    for i, length in enumerate(lengths):
        output[i, length:] = 0
    return output
```

The linear projection output at each position is passed through argmax to select the most likely character:

```python
predictions = outputs.argmax(dim=-1)
```

The predicted token IDs are converted back to characters using the tokenizer's vocabulary, excluding special tokens:

```python
results = ["".join(tokenizer.convert_ids_to_tokens(x, skip_special_tokens=True)) for x in pred]
```

### 6.2 Performance

On CPU, the model evaluates all 2,643 test samples in approximately 149 seconds - well within the 250-second limit. This efficiency is attributable to three factors:
1. The lightweight GRU architecture (no attention, no transformer).
2. Conditional use of `pack_padded_sequence` for long sequences.
3. Batch processing with size 32.

### 7.2 Ablation: Contribution of Each Phase

While a formal ablation study was not conducted in the original solution, we can reason about each phase's contribution:

- **Without Phase 1 (autoencoding pretrain):** The model begins Phase 2 with random embeddings and hidden states. Given only 15 epochs of chunked training, the model would likely converge to a higher loss valley. Autoencoding provides a warm start that steers optimization toward linguistically meaningful representations.
- **Without Phase 3 (full-length fine-tuning):** The model would have only seen 256-character chunks. At test time, it would encounter sequences 3 to 16 times longer than anything seen during training, leading to poor generalization on longer sequences.
- **Without chunking (Phase 2 only):** This is essentially the baseline approach with a larger model and more training epochs. It would likely underperform the full solution by a significant margin.

## 8. Discussion

### 8.1 Why This Approach Works

The solution's effectiveness can be attributed to three principles:

**Locality bias.** The model architecture (bidirectional GRU with local 256-character chunks) strongly biases learning toward local character patterns. This is appropriate for autocorrect, where most errors are character-level transpositions, substitutions, or insertions that depend on at most a few surrounding characters.

**Curriculum learning.** The three-phase training strategy follows a principled curriculum: autoencoding -> chunked correction -> full-length correction. Each phase introduces more complexity while building on previously learned representations.

**Computational awareness.** Every design decision - from the choice of GRU over LSTM to the conditional use of pack_padded_sequence - reflects the CPU-only, time-limited evaluation constraint. The solution is not the most complex possible, but it is the most efficient for the given constraints.

### 8.3 Potential Improvements

Several directions could further improve performance:

- **Synthetic error augmentation:** Generating additional training pairs by simulating common typographical errors (keyboard adjacency, character omission) could improve robustness.
- **Beam search decoding:** Instead of greedy argmax at each position, beam search could consider multiple correction paths, though this would increase inference time.
- **Transformer with local attention:** A transformer using a sliding window (e.g., 128-character local attention) could match the locality bias while enabling better parallelization during training.
- **Ensemble:** Averaging predictions from multiple GRU models trained with different random seeds typically reduces CER by 5 to 10 percent.

## 9. Conclusion

This editorial presented a solution to the AICC Round 1 Autocorrect task that achieves a CER of 0.0362, substantially outperforming the baseline of 0.0944. The solution's core innovations - chunked training, curriculum learning through three phases, and a bidirectional GRU architecture - collectively exploit the local nature of character-level error correction while respecting strict computational constraints. The solution serves as a case study in designing efficient neural architectures for resource-constrained NLP tasks, demonstrating that thoughtful data and training strategies can yield competitive results without large pre-trained models.
